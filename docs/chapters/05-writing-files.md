## 5.1 ファイルに「書く」とはどういうことか

第4章では「ファイルを読む」を学びました。この章ではいよいよ「ファイルに書く（作成・編集・整形）」に進みます。

Claude Code がファイルを書くとき、2つのパターンがあります：

| パターン | 説明 | 例 |
|----------|------|-----|
| 新規作成 | 存在しないファイルを新しく作る | 「提案書の下書きを proposal-draft.md として保存」 |
| 既存編集 | 今あるファイルの内容を変更する | 「proposal.txt の第2段落を書き直して」 |

どちらの場合も、Claude Code はファイルを変更する前に確認を求めます（第3章参照）。「Y」を押して許可すると、実際にファイルが変更されます。

---

## 5.2 新しい文書を作らせる

### 基本の形：「内容 + 保存先ファイル名」を指定する

```
> 以下の条件で提案書の下書きを作って、proposal-draft.md として保存して：
  - クライアント：株式会社サンプル
  - 提案内容：Webサイトリニューアル
  - 費用：別途見積もり（金額は書かない）
  - 文量：A4用紙1〜2枚程度
  - 読み手：経営者（専門用語は使わない）
```

Claudeが確認を求めてきたら「Y」を押して許可します。

### テンプレートから量産する

既存のひな型ファイルを参考に複数の文書を作る場合：

```
> template-invoice.txt を参考に、以下のクライアント向けの
  請求書を作って invoice-20250430-yamada.txt として保存して：
  - 宛名：山田太郎様
  - 件名：Webデザイン制作費
  - 金額：150,000円（税込）
  - 支払期限：2025年5月31日
```

テンプレートのフォーマットを維持しながら、差し込む情報だけ変えてくれます。

## 5.3 既存ファイルを編集させる

### 文章の一部を書き直す

```
> proposal.txt を読んで、3段落目を「もっと具体的な効果・数字を入れて」
  書き直して。それ以外の部分は変えないこと。
```

「それ以外の部分は変えないこと」という制約を加えることで、意図しない変更を防げます。

### 全体のトーンを変える

```
> draft.txt を読んで、全体のトーンを「丁寧なビジネス文書」から
  「フレンドリーなカジュアル文体」に書き直して。
  内容（事実・数字）は変えないこと。
  書き直した内容は同じファイルに上書き保存して。
```

### 複数ファイルを一括整形

```
> invoices/ フォルダにある全ての .txt ファイルを読んで、
  各ファイルの「支払期限」の日付フォーマットを
  「令和X年X月X日」から「YYYY年MM月DD日」の形式に統一して保存して。
```

## 5.4 実践レシピ集

### レシピ1：週次レポートを自動生成

週の作業ログや記録から週次レポートを作る：

```
> daily-logs/ フォルダの今週分のファイルを全部読んで、
  以下の構成で週次レポートを作って weekly-report-2025W17.md として保存して：

  # 今週の業務報告（2025年4月21日〜25日）

  ## 完了した案件
  ## 進行中の案件と進捗
  ## 来週の予定
  ## 困っていること・相談事項

  各項目は箇条書きで、1項目につき30字以内に収めること。
```

### レシピ2：提案書・企画書のたたき台を瞬時に作る

```
> 以下の情報から提案書のたたき台を作って proposal-kato-20250425.md として保存して：

  背景：加藤様（飲食業）は集客が伸び悩んでいる
  提案したいこと：SNSを活用したブランディング支援
  想定期間：3ヶ月
  料金：月額 80,000円

  構成：
  1. 現状の課題（仮説）
  2. 提案の概要
  3. 進め方（月別ロードマップ）
  4. 期待できる効果
  5. 料金・お支払いについて

  文体：ですます調、丁寧かつ読みやすく
```

### レシピ3：メールの文面を量産する

同じ種類のメールを複数のクライアント向けに作る：

```
> 以下のテンプレートを使って、clients.txt に書かれた各クライアントへの
  お礼メールを作って。メールはそれぞれ別ファイル（mail-〇〇.txt）として保存して：

  テンプレート：
  ----
  [宛名]様
  本日はお時間をいただきありがとうございました。
  [打ち合わせ内容の一言メモ] についてお話しできて、大変参考になりました。
  引き続きよろしくお願いいたします。
  [自分の名前]
  ----

  clients.txt の形式：
  山田太郎｜新サービスの方向性確認
  鈴木花子｜年間契約の更新について
```

### レシピ4：既存資料の品質チェックと修正

```
> contract-draft.txt を読んで、以下の観点でチェックして問題点をリストアップして：
  1. 意味が曖昧な表現
  2. 法的リスクになりうる記述
  3. 抜けている可能性がある項目

  チェック結果を確認後、問題があれば修正版を contract-revised.txt として保存して。
```

## 5.5 Claude が「間違えた」ときの対処法

AIが文書を生成しても、内容が意図と違うことがあります。対処法を知っておきましょう。

### 修正指示を続けて出す

一度のやり取りで完璧を求めず、フィードバックを繰り返すのが効果的です：

```
> 良いですが、結論の部分をもう少し前向きなトーンに変えて。
> 第2段落の「困難」という言葉を使わないように書き直して。
> 全体的に少し短く、A4半枚に収まる分量にして。
```

### 大きく間違えた場合はGitで元に戻す

ファイルの内容が大幅にズレた場合：

```bash
# ファイルを変更前の状態に戻す
git restore .
```

その後、より明確な指示を出し直します。

### 「良い部分だけ取り出す」指示

```
> 今作ってもらった下書きの中で、第1段落と第3段落は良い。
  第2段落だけ書き直して。
```

全部やり直さなくても、部分的な修正指示を出せます。

## 5.6 大切な習慣：作業前のセーブ

重要なファイルを編集させる前には、必ずGitでセーブをしましょう：

```bash
# 作業前のセーブ（1行で実行できます）
git add . && git commit -m "提案書編集前のバックアップ"
```

万一の場合は `git restore .` で元に戻せます。

---

## ここで詰まったら

| 症状 | 対処法 |
|------|--------|
| ファイルが保存されなかった | 確認プロンプトで「Y」を押したか確認。もう一度「〇〇として保存して」と指示 |
| 意図しない変更がされた | `git restore .` で元に戻す |
| 文章のトーンが毎回違う | CLAUDE.md に文体の好みを書いておく（第6章） |
| ファイルが長くなりすぎた | 「〇〇字以内で」「A4一枚相当の分量で」と制約を追加 |

---

## 5.7 この章のまとめ

- ファイルへの書き込みは「内容 + 保存先ファイル名」の指定がポイント
- 既存ファイルの編集は「〇〇以外は変えないこと」という制約が重要
- 一度で完璧を求めず、フィードバックを重ねるのが効果的
- 重要ファイルを編集する前は必ず `git commit` でバックアップ
